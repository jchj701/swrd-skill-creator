# SWRD 设计思路溯源

> Source: 本次 brainstorming 会话
> Collected: 2026-05-23

## 背景

SWRD 的设计不是凭空产生的，而是基于对 5 个项目的深度调研和横向对比。每个项目贡献了不同的设计理念。

## 五个项目的贡献

### SkillFather（工程化骨架）
贡献了最完整的工程化基础：标准化目录结构、状态机、Eval 驱动开发、可观测性设计。
SWRD 直接继承了 SkillFather 的目录规范和状态机框架，并在此基础上扩展。

### darwin-skill（评分优化）
贡献了 8 维度评分体系和 hill-climbing 优化循环。
SWRD 改进为 3 子 Agent 投票 + 动态阈值，解决了"自己改自己评"的偏差问题。
同时吸收了 Runtime 适配性审查（红灯/绿灯扫描）。

### Karpathy autoresearch（棘轮机制）
贡献了最核心的"约束即解放"思想：单文件修改、固定时间预算、单一标量指标、git 即记忆。
SWRD 将"永不停止"改为"人在回路"，因为 Skill 质量比 loss 更微妙，需要人类判断。

### Claude Autoresearch（模块化路由）
贡献了 95% token 缩减的模块化架构：薄路由 + 多个独立命令文件。
SWRD 借鉴了"多种循环模式"的思想（metric/saturation/hypothesis/refinement），但未直接采用其 13 命令体系，因为 SWRD 的定位是元 Skill 而非通用工具集。

### XSkill（经验检索思想）
贡献了"从轨迹中自动提取经验"的思路。
SWRD 未直接实现 embedding 检索（工程复杂度高），但通过 karpathy-llm-wiki 实现了轻量级的经验记录和检索。

## 关键设计决策

### 为什么选择人在回路而不是全自动？
darwin-skill 和 Karpathy autoresearch 都是全自动的，但 Skill 质量比 ML 模型的 loss 更微妙：
- loss 是单一标量，好坏一目了然
- Skill 质量涉及结构、效果、可读性、跨平台兼容等多个维度
- 有些改进在分数上可能不显著，但对用户体验提升很大（反之亦然）
所以 SWRD 选择"每个模块优化后暂停，用户确认再继续"。

### 为什么选择 YAML + JSONL 双日志格式？
- YAML：人类可读，适合直接查看和手动编辑
- JSONL：机器可解析，每行一个独立 JSON，适合作为 ML 训练集
- 双格式互补，不需要在"可读性"和"可训练性"之间做取舍

### 为什么选择 karpathy-llm-wiki 而不是 embedding 检索？
- karpathy-llm-wiki 零依赖、纯文件、完全离线可用
- embedding 检索需要向量数据库和 embedding 模型，增加部署复杂度
- 对于 Skill 经验记录这种场景，关键词检索 + 结构化目录已经足够
- 未来可以升级：当 wiki 规模增长到一定程度，再引入 embedding 检索

### 为什么模块化约束是 ≤3 文件/模块、≤300 行/文件？
- 3 个文件：一个模块通常包含定义、实现、测试三个文件
- 300 行：一个文件超过 300 行意味着职责可能不单一
- 这两个数字来自实践经验，不是理论推导
- 约束的目的是强制模块化思考，不是制造障碍

## 被否决的方案

### 否决：全自动无人干预
理由：Skill 质量需要人类判断，有些改进在分数上不显著但对体验提升很大。
替代方案：人在回路，每个模块优化后暂停。

### 否决：单一日志格式（仅 JSON 或仅 YAML）
理由：JSON 可读性差，YAML 不适合做训练集。
替代方案：YAML 主日志 + JSONL 训练集双格式。

### 否决：llm-wiki 作为外部依赖
理由：创建 Skill 的环境可能离线，无法联网安装。
替代方案：内置零依赖引擎，遵循 karpathy-llm-wiki 规范，未来可无缝迁移。

### 否决：联动工程假设为 git 仓库
理由：被 Skill 操作的"工程"不一定是代码，可能是一个目录、一组文件。
替代方案：通用适配层（git/非 git/自动 init/全量备份）。

### 否决：embedding 向量检索
理由：工程复杂度高，对 Skill 经验记录场景来说过度设计。
替代方案：karpathy-llm-wiki 的关键词检索 + 结构化目录。
