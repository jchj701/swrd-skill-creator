# 用户提出的关键约束

> Source: 本次 brainstorming 会话中的用户输入
> Collected: 2026-05-23
> Published: 2026-05-23

## 版本号约束

"每次迭代棘轮进化时，版本号应该具备升级能力。使用三段式版本号，从0.1.0作为首个版本，除非用户明确更新大版本、中版本，否则一定更新最小版本。整个skill只需要一个地方写当前的版本号SKILL.md，变更记录与变更历史单独写一个文件。禁止回退版本号。"

**设计实现**：
- 版本号唯一位置：SKILL.md frontmatter
- 变更记录：CHANGELOG.md
- 棘轮前进自动升 patch，用户可手动升 minor/major
- git revert 不改变版本号

## 全日志约束

"我期望越细越好，仅记录在一个文件中，做到这个skill创建后，一旦被使用就能持续记录，不被覆盖，跨任务也要追加。日志除了记录调用的相关情况，调用结果也要记录，我期望用于二次训练进化时，能够辅助判断，所以格式需要考虑全面，可能会用于训练集、验证集。"

**设计实现**：
- YAML 主日志（人类可读，追加写入）
- JSONL 训练集（机器可解析，每行独立 JSON）
- 记录粒度：每次调用、每次 Eval、每次优化迭代、每次回退

## 模块化约束

"我期望范围需要小。如果模块化设计具备高内聚、低耦合的成熟度，那么是可以的，是否能从skill创建时就约束并设计好。"

**设计实现**：
- 每个模块 ≤ 3 文件、≤ 300 行
- 创建时通过 3 子 Agent 独立审查
- 审查不通过回退重新拆分

## 离线可用约束

"创建skill的环境可能离线环境，无法联网，也要做到通用支持。"

**设计实现**：
- karpathy-llm-wiki 零依赖、纯文件、完全离线可用
- 内置 wiki 引擎，不依赖外部服务
- 遵循 karpathy-llm-wiki 规范，联网后可无缝迁移

## 联动工程通用性约束

"联动的工程不一定是代码，或者非git仓库，需要做到通用支持。目标是用git来管理，可以进行手动init；要足够通用。"

**设计实现**：
- 通用适配层：git 仓库 / 非 git 目录 / 自动 init / 全量备份
- 实验前快照，实验后恢复
- 用户可手动 git init

## 评分独立性约束

"需要有独立的专家进行自动审核，因为你自己检查可能因为上下文原因导致问题。"

**设计实现**：
- 3 子 Agent 独立评分，取中位数
- 子 Agent 不共享主 Agent 的上下文
- 子 Agent 超时时降级为主 Agent 自检

## 棘轮粒度约束

"每次变动需要参考darwin-skill需要单个文件进行调整。每次棘轮的递进范围需要足够可控，允许测试轮数增加，但是要做的每一轮都可测可回溯。"

**设计实现**：
- 每次只改一个模块（1-3 个文件）
- 每次修改前 git tag 快照
- 每次修改后独立评分
- keep/revert 决策基于评分对比
- 所有操作记录到 telemetry
